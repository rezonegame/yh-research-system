# Lightweight Local Operations Reference

Use this reference when the project asks for few staff, few devices, local cultural content, cultural-tourism activation, merchant participation, resident memory, return loops, or low-maintenance implementation.

## Operating Principle

The best lightweight system is not the smallest idea; it is the smallest reliable operating system that still carries local meaning.

Prefer:

- paper passport, route card, stamp, sticker, QR explanation page;
- one-sentence merchant participation;
- public-space tasks that can be completed without staff intervention;
- monthly content refresh rather than permanent hardware;
- local memory collection with consent;
- route loops that bring visitors back to public nodes and participating businesses.

Avoid:

- app-first systems without maintenance budget;
- AR or sensors when a sign, card, or facilitator can do the job;
- tasks that require merchants to explain complex rules;
- competitive mechanics that discourage elders, families, or casual visitors;
- fictional resident stories;
- uncontrolled crowding near heritage or residential edges.

## Lightweight Game Kit

| Component | Low-burden format | Planning / operation value |
|---|---|---|
| Passport | Folded paper card | Route completion, return loop, low-tech record |
| Stamp | Physical stamp or sticker | Completion signal without app dependency |
| QR page | Static page or form | Deeper story, feedback, optional data |
| Task card | A6 card or table tent | Merchant-light participation |
| Route board | Reusable board at entry or public node | Orientation and crowd distribution |
| Memory card | Consent-aware interview card | Resident memory without extraction |
| Monthly prompt | One rotating question | Repeat visits and fresh local content |
| Exchange rule | Small non-monetary reward | Return to node/business without hard selling |

## Staff Model

| Role | Minimum load | Notes |
|---|---:|---|
| Coordinator | 0.2-1 FTE | Updates route, replaces materials, talks to merchants |
| Weekend helper | 1-2 people during peaks | Only for orientation and issue handling |
| Merchant point | 10-15 pilot merchants | One simple action each: stamp, display card, or provide phrase |
| Community liaison | part-time / event-based | Resident consent, memory collection, conflict feedback |

## Merchant Participation Levels

| Level | Merchant burden | Suitable action |
|---|---|---|
| A: display only | very low | Place task card / QR / route sign |
| B: stamp point | low | Give stamp after visitor completes prompt |
| C: micro-benefit | medium | Small item, discount, story phrase, limited route product |
| D: co-creator | higher | Host memory night, workshop, mini exhibition, local product |

Start with A/B/C. Use D only with willing merchants and clear benefit.

## Return-Loop Design

Use at least one return-loop:

1. Spatial return: routes end at a public node, service desk, or exchange point.
2. Temporal return: monthly route theme or seasonal card.
3. Merchant return: completed route unlocks a low-pressure benefit.
4. Memory return: visitors/residents contribute prompts for next month.
5. Governance return: feedback enters a visible improvement log.

## Resident Memory Rules

- Do not invent memory.
- Do not publish identifiable stories without consent.
- Separate personal privacy, relocation conflict, family history, and public local knowledge.
- Let residents choose whether a memory becomes a private note, anonymized quote, public display, or design requirement.
- Use silence as data: what residents do not want to share is also a boundary.

## Carrying-Capacity Triggers

Watch for queues blocking entrances, photo-taking near residences, night noise, merchant complaints, heritage wear, route abandonment, and residents avoiding the area during events.

When triggered, reduce route density, shift time windows, move exchange points, cap group size, or remove the task.
